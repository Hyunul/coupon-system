# HikariCP 풀 크기 실험 — "풀은 클수록 좋은 게 아니다"

> 날짜: 2026-07-21 · Phase: 1 · 커밋: `c508174` (V3 인덱스 추가 전) · 실행: `scripts/hikari-experiment.sh "5 20 100" 1000 2m`

## 1. 가설

단일 행 락에 직렬화되는 워크로드에서 커넥션 풀을 키우면 (a) 대기 지점만 Tomcat→풀→DB로 이동하고 (b) DB 안 락 경합이 늘어 오히려 나빠질 것이다. 최적점은 생각보다 작을 것.

## 2. 환경

baseline 리포트와 동일 머신/설정. 변인은 `hikari.maximum-pool-size`(5/20/100)뿐. 각 런마다 앱 재시작(JVM 콜드 스타트 조건 동일), 재고 10,000 재시드. 부하는 `issue-baseline.js` constant-arrival-rate **1,000rps × 2분** (baseline의 5,000rps는 전 구간 붕괴라 변별력이 없어 1,000rps 선택).

## 3. 결과

재고 1만 장은 시작 후 수십 초 내 소진되므로, 이후 구간은 409(SOLD_OUT) 응답이 지배한다. 409도 현재 구현에선 `FOR UPDATE` 행 락을 통과한다는 점에 주의.

| 지표 | pool=5 | pool=20 | pool=100 |
|---|---|---|---|
| 완료 요청 (120k 목표) | 70,204 | 71,411 | 73,161 |
| dropped iterations | 49,797 | 48,590 | 46,840 |
| **응답 중앙값** | 1,130ms | **91ms** | 1,077ms |
| 응답 p95 | 18.1s | 18.1s | 17.5s |
| 서버측 p99 | 1.76s | **0.234s** | 1.41s |
| hikari active 최대 | 5/5 | 20/20 | 100/100 |
| hikari pending 최대 | 194 | 179 | 99 |
| 발급(201) 중앙값 | 17.8s | 18.0s | 17.4s |
| 정합성 | 0/0/0 | 0/0/0 | 0/0/0 |

## 4. 분석

- **pool=5 (기아)**: 커넥션이 병목의 최전선. 스레드 194개가 풀 대기 → 모든 응답이 대기 시간을 떠안아 중앙값 1.1초.
- **pool=20 (최적)**: 중앙값 91ms, 서버 p99 0.23s로 **다른 두 조건 대비 5~12배** 우위. 12코어 머신에서 HikariCP 공식(2×코어)과 대략 일치.
- **pool=100 (과대)**: 풀 대기는 줄었지만(pending 99) 커넥션 100개가 동시에 DB에 진입해 **같은 행 락을 100개가 경합** → 락 큐 + 컨텍스트 스위칭 비용으로 중앙값이 다시 1초대로 악화. "대기가 풀 앞에서 DB 안으로 이동했을 뿐"이라는 가설 그대로.
- 발급(201) 응답이 세 조건 모두 ~18s인 것은 재고 소진 직전 구간의 락 큐 길이가 풀 크기와 무관하게 arrival rate로 결정되기 때문 — 근본 원인(직렬화)은 풀 튜닝으로 해결되지 않음을 보여준다.

## 5. 다음 액션

- 기본값을 pool=20으로 유지 (application.yml에 명시).
- 락 직렬화 자체를 없애는 Phase 2(Redis 원자 연산)에서 동일 매트릭스를 재실험하면 "병목이 사라진 뒤 풀 크기 민감도가 어떻게 변하는가"라는 후속 콘텐츠가 됨.
