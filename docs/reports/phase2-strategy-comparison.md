# 재고 차감 3전략 실측 비교 — 비관적 락 vs Redisson 분산락 vs Redis Lua

> 날짜: 2026-07-22 · Phase: 2 · 시나리오: `issue-spike.js` (100→5,000rps, 재고 10,000, 30,000+ 유니크 사용자) · 실행: `scripts/strategy-compare.sh`

## 1. 가설

임계 경로에서 MySQL 행 락을 제거하면 처리량이 크게 오를 것이다. 단, Redisson 분산락은 "락 위치를 DB에서 Redis로 옮긴 것"이라 병목이 재현되고, Lua 원자 스크립트만이 락 없는 판정으로 유의미한 개선을 보일 것이다 (roadmap 4.4).

## 2. 환경

| 항목 | 값 |
|---|---|
| 커밋 | Phase 2 구현 커밋 (3전략 동일 코드베이스, V3 인덱스 포함) |
| 전략 선택 | `--coupon.issue.strategy=pessimistic\|redisson\|lua` (재빌드 없음, 런마다 JVM 콜드 스타트) |
| 머신 | Windows 10, 12코어/32GB, k6+SUT 동일 머신 — **상대 비교 목적** |
| 고정 조건 | Tomcat threads.max=200, Hikari pool=20, Redis 단일 노드(compose) |
| 시드 | SQL 시드 후 `PATCH /status OPEN` → Redis 재고 초기화 경로 통과 |

## 3. 결과 (k6 클라이언트측 = 비교의 기준)

| 지표 | ① pessimistic | ② redisson | ③ lua |
|---|---|---|---|
| 총 요청 | 762,446 | 761,995 | 759,985 |
| **정상 처리 (201/409)** | 109,823 (14.4%) | 54,350 (7.1%) | **336,727 (44.3%)** |
| **발급(201) 응답 중앙값** | 12,042ms | 9,421ms | **34ms** |
| 발급(201) p95 | 44,391ms | 34,159ms | **1,423ms** |
| TCP 연결 거부 | 652,524 | 707,493 | **423,159** |
| 60s 타임아웃 | 99 | 125 | 99 |
| DB 발급 수 / 중복 | 10,000 / 0 | 10,000 / 0 | 10,000 / 0 |
| Redis 재고 / 발급자 수 | (미사용) | 0 / 10,000 | 0 / 10,000 |
| 서버측 (Prometheus) | 신뢰 불가* | 신뢰 불가* | p99 0.71s, tomcat busy 200, hikari pending 160 |

\* **관측의 함정 발견**: pessimistic/redisson 런에서는 폭주 구간에 Prometheus 스크레이프 요청 자체가 TCP 거부에 막혀(다른 요청들과 같은 운명), 서버측 지표가 평온 구간만 반영됐다(tomcat busy가 2로 기록되는 식). 서버가 살아있던 lua 런만 서버측 지표가 유효하다. **관측 시스템도 장애 도메인 안에 있다** — 이 자체가 Phase 4 장애 훈련의 소재다.

## 4. 분석

- **② redisson이 ①보다 나빴다 (정상 처리 절반)**: 매 요청 락 획득/해제 왕복 + `tryLock` 대기(최대 5s)가 Tomcat 스레드를 점유한다. 재고 소진 후에도 모든 요청이 락 큐를 통과해야 409를 받는다. 락 병목이 DB에서 Redis로 이동했을 뿐이라는 가설 그대로.
- **③ lua는 발급 응답 12s → 34ms (354×), 정상 처리 3.1×**: 판정이 Redis 단일 원자 연산(마이크로초대)이라 락 대기가 없다. 재고 소진 후 409는 Caffeine 메타 캐시 + Lua 호출만으로 응답되어 DB를 전혀 건드리지 않는다.
- **③도 TCP 거부 42만 건이 남은 이유**: 발급 성공 구간에서 이력 INSERT가 여전히 동기라 Tomcat 스레드 200개가 DB에 블록되고(hikari pending 최대 160), 5,000rps는 단일 머신(k6 공존) 수용량도 초과한다. **다음 병목은 "동기 DB 기록"** — Phase 3의 Redis Stream + 워커 분리 동기가 수치로 확보됐다.
- 정합성은 3전략 모두 초과 0/중복 0, Redis-DB 대사 일치(10,000=10,000). Lua의 정합성은 Testcontainers 동시성 테스트(300명→100장)로도 CI에서 상시 검증된다.
