# Phase 1 Baseline — 5,000rps 스파이크에서 비관적 락 MVP가 무너지는 모양

> 날짜: 2026-07-21 · Phase: 1 · 커밋: `c508174` (V3 인덱스는 이 측정 이후 추가 — explain 리포트 참조)

## 1. 가설

`SELECT ... FOR UPDATE`로 이벤트 행 하나에 모든 발급을 직렬화하는 구조는 5,000rps 스파이크를 감당하지 못할 것이다. 어디서(스레드? 풀? 커널?) 먼저 무너지는지 위치를 확인한다.

## 2. 환경

| 항목 | 값 |
|---|---|
| 커밋 해시 | `c508174` (앱 코드는 V3 제외 동일) |
| 머신 | Windows 10, 12 논리코어, 32GB RAM (부하 생성기·SUT 동일 머신 — **상대 비교 목적**) |
| 앱 | 호스트 JDK 21, Tomcat threads.max=200(기본), HikariCP pool=20, timeout 30s(기본) |
| 인프라 | Docker Desktop(WSL2): MySQL 8.0(13306), 데이터는 named volume |
| 시드 | event 1, 재고 10,000, OPEN |

## 3. 시나리오

`k6/scenarios/issue-spike.js` — ramping-arrival-rate 100rps(30s) → 5,000rps(10s 램프) → 5,000rps 유지(2m) → 100rps(1m). maxVUs 10,000. 종료 후 `verify-consistency.sql`.

## 4. 결과

| 지표 | 값 |
|---|---|
| 시도 요청 | 585,177 (2,340/s) + **dropped iterations 196,220** (k6가 스케줄조차 실패) |
| 실패율 (k6) | **99.15%** — connection refused 552,791건, 60s 타임아웃 12,470건 |
| 서버에 도달한 요청 | 37,157건 (전체 시도의 6.4%) |
| 서버측 p99 (도달분) | 17.68s |
| 발급 성공(201) 응답시간 | p95 41.61s |
| `tomcat.threads.busy` 최대 | **200 / 200 (포화)** |
| `hikaricp.connections` | active 최대 20/20, **pending 최대 178**, timeout 14건 |
| 정합성 (초과/중복/대사) | **0 / 0 / 0** — 정확히 10,000건 발급 |

## 5. 분석

붕괴의 사슬이 계층별로 명확하다:

1. 모든 발급이 이벤트 행 하나의 락에 직렬화 → 처리 속도가 락 처리량에 상한.
2. Tomcat 스레드 200개가 전부 "락/커넥션 대기"로 블록 → 새 요청을 받을 스레드가 없음.
3. 커넥션 풀 20개는 전부 active, 나머지 178 스레드가 풀 대기.
4. accept 큐(백로그)까지 차면 OS가 TCP 연결을 거부 → **실패의 95%는 애플리케이션이 아니라 TCP 레벨에서 발생**.

즉 "DB가 느리다"가 아니라 "락 직렬화 + 스레드/풀 병목이 연쇄"라는 구조 문제다. 그 와중에 정합성이 0/0/0인 것은 비관적 락의 유일한 장점 — 정확하지만 느리다.

주의: 부하 생성기와 SUT가 같은 머신이라 절대 수치는 보수적으로 볼 것. 5,000rps 구간에서 k6 자체도 CPU를 소모한다.
