# Phase 4 회고 — 저수준 튜닝: 증거가 로그에 없을 때 어디를 봐야 하는가

> 2026-07-22 · 커밋 범위: `4f27114`~`ed295db` · 근거: [GC 리포트](../reports/phase4-gc-tuning.md) · 포스트모템 [풀 고갈](../postmortems/drill1-connection-pool-exhaustion.md) / [타임아웃 미설정](../postmortems/drill2-missing-timeout.md) / [maxclients](../postmortems/drill3-redis-maxclients.md)

## 1. 배경과 목표

[로드맵 Phase 4](../coupon-system-roadmap.md)의 목표는 두 가지입니다: GC 로그를 켜고 STW가 p99에 만드는 스파이크를 잡는 것, 그리고 계획된 장애 훈련 — 장애를 일부러 만들고 **패킷·스레드덤프 수준의 증거**로 원인을 규명한 뒤 포스트모템을 남기는 것. "장애 대응 경험"을 수치와 증빙으로 만드는 단계입니다.

## 2. 문제 상황

### 2.1 p95 158ms의 범인은 "큰 pause"가 아니었다

1,000rps 부하에서 기본 G1의 p95는 158ms였습니다. GC 로그를 열어보니 개별 pause는 최대 8.15ms로 무해해 보였는데, **횟수가 2분에 191회**였습니다. 기본 설정의 JVM이 작은 heap에서 출발해 부하 중 heap 확장과 young 영역 재조정을 반복한 것 — 큰 pause 한 방이 아니라 잦은 pause와 리사이징의 공진이 꼬리를 만들었습니다.

### 2.2 로그가 침묵하는 장애들

세 훈련 모두 공통점이 있었습니다: **애플리케이션 로그만 보면 원인을 알 수 없다**.

- 풀 고갈(훈련 1): 지연 3.6초, 하지만 에러 로그 0줄 (Hikari 타임아웃 30s 도달 전).
- 타임아웃 미설정(훈련 2): 지연 5,000배, 에러율 0% (예외를 삼키는 코드).
- maxclients(훈련 3): 아예 기동 불능 — 부하 테스트가 아니라 배포가 실패하는 형태.

부수 트러블 둘: 프로젝트 경로의 공백이 `-Xlog:gc*:file=<절대경로>`를 깨뜨려 JVM이 기동 실패했고(상대 경로로 해결 — Phase 0에서 경고했던 리스크가 실제로 문 것), 반복 부하 후 Docker Desktop 엔진이 내려가 인프라를 재기동해야 했습니다.

## 3. 해결 과정

### 3.1 GC 설정 비교 (실측)

| 선택지 | pause (2분) | k6 p95 | 판단 |
|---|---|---|---|
| G1 기본 (heap 동적) | 191회, 합 368ms | 158.1ms | ✗ 리사이징이 주범 |
| **G1 + heap 2g 고정 + MaxGCPauseMillis=50** | 26회, 합 95ms | **2.6ms** | **✓ 개선의 9할, 기본 채택** |
| ZGC generational + 2g | 11회, 최대 **0.03ms** | 2.2ms | △ pause는 소멸했지만 추가 이득 0.4ms — Phase 5 재평가 |

"더 좋은 GC로 바꾸기" 전에 "GC가 일할 필요를 줄이기(heap 고정)"가 먼저였습니다.

### 3.2 증거 수집 도구 선택 (Windows 호스트 + 컨테이너 인프라)

| 선택지 | 장점 | 단점 | 채택 |
|---|---|---|---|
| Windows pktmon/Wireshark | 호스트 루프백 캡처 가능 | 필터 문법 상이, 자동화 어려움 | ✗ |
| 컨테이너에 tcpdump 직접 설치 | 단순 | 이미지 오염, 재현성 낮음 | ✗ |
| **netshoot 사이드카** (`--net container:<이름>`) | 대상 컨테이너 네트워크를 무설치로 캡처, 스크립트화 용이 | 이미지 풀 필요 | **✓** |
| 스레드덤프 (`jcmd Thread.print`) | 호스트 JVM 내부는 패킷보다 직접적 | 순간 스냅샷 | **✓ (훈련 2)** |

### 3.3 각 훈련의 결정적 증거

- **훈련 1**: 부하 60초간 MySQL로 향한 **신규 SYN 1건** + `hikaricp.connections.pending=197` → "DB가 느린 게 아니라 풀 앞에서 대기"를 패킷의 *부재*로 증명.
- **훈련 2**: 스레드덤프에서 Tomcat 스레드 200개 전원이 `Mono.block`(ReactorClientHttpRequest)에 파킹 → `RestClient.create()`가 클래스패스의 reactor-netty를 자동 선택했고 **그 기본 readTimeout 10초가 실질 용량(200÷10s=20rps)을 결정**하고 있었음을 발견. "타임아웃 미설정 = 모르는 기본값에 위임".
- **훈련 3**: `rejected_connections` 0→4 + `ERR max number of clients reached` 로그 → Redisson fail-fast로 기동 불능. `CONFIG SET`(무중단) + 앱 재기동의 2단계 복구까지 검증 (복구 후 3,001/3,001 정상, 중앙값 10ms).

## 4. 결과

| 항목 | Before | After |
|---|---|---|
| 1,000rps p95 (GC) | 158.1ms (pause 191회) | **2.6ms** (pause 26회, heap 고정) |
| 산출물 | — | GC 리포트 1편 + **포스트모템 3편** (증거 파일 포함) |
| 미수행 (정직하게) | — | 훈련 2(TIME_WAIT 폭증)·훈련 5(패킷 유실)는 Nginx/tc 환경이 갖춰지는 Phase 5로 이월. NewRatio 실험은 변별력 소실로 생략 |

## 5. 배운 점

- 기본 G1의 p95 158ms를 GC 로그로 추적해 원인이 pause 크기(8ms)가 아니라 빈도(191회)와 heap 리사이징임을 확인했고, heap을 고정했더니 p95가 2.6ms가 되어 "GC 교체보다 heap 고정이 먼저"임을 수치로 배웠습니다.
- 커넥션 풀 고갈을 패킷으로 증명했습니다 — 부하 60초간 DB로 향한 신규 연결 시도가 1건뿐이라는 것(증거의 부재가 증거)이 "DB가 느리다"는 오판을 뒤집는 결정적 근거였습니다.
- 타임아웃을 명시하지 않으면 서드파티의 기본값(reactor-netty 10초)이 시스템 용량을 결정한다는 것을 스레드덤프 200개의 동일한 파킹 프레임으로 확인했습니다.

## 6. 다음 Phase 예고

로컬에서 만들 수 있는 증거는 다 만들었습니다. Phase 5는 실배포입니다 — Nginx 뒤 앱 2대, 무중단 배포, 부하 중 인스턴스 강제 종료(chaos), 그리고 이번 포스트모템들이 약속한 Prometheus 알림 규칙(pending, rejected_connections, 지연 기반 알림)을 실제로 거는 단계입니다. 이월된 TIME_WAIT·패킷 유실 훈련도 그 환경에서 수행합니다.
