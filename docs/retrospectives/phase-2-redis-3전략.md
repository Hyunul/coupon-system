# Phase 2 회고 — 재고 차감 3전략 실측: 락을 옮기는 것과 없애는 것의 차이

> 2026-07-22 · 커밋 범위: `5ca4dfc`~`62398ae` · 근거 리포트: [3전략 비교](../reports/phase2-strategy-comparison.md)

## 1. 배경과 목표

Phase 1의 결론은 "비관적 락은 정확하지만, 이벤트 행 하나에 모든 발급이 직렬화되어 5,000rps에서 실패율 99.15%"였습니다. Phase 2의 목표는 [로드맵 4.4](../coupon-system-roadmap.md)의 재고 차감 3전략 — ① MySQL 비관적 락, ② Redis 분산락(Redisson), ③ Redis Lua 원자 스크립트 — 을 **전부 구현해서 동일 시나리오로 실측 비교**하는 것입니다. 비교가 가능했던 것은 Phase 1에서 발급 로직을 `IssueStrategy` 인터페이스 뒤에 숨겨둔 덕분으로, 전략 교체는 `--coupon.issue.strategy=lua` 플래그 하나로 끝났습니다.

## 2. 문제 상황

### 2.1 분산락이 비관적 락보다 느렸다

②를 구현하고 스파이크를 걸자 예상보다 나쁜 결과가 나왔습니다. 정상 처리(201/409)가 54,350건 — **비관적 락(109,823건)의 절반**. 락을 DB에서 Redis로 옮겼는데 왜 더 나빠졌을까요.

원인은 락의 위치가 아니라 **락의 존재 자체**였습니다. 매 요청이 락 획득/해제로 Redis 왕복 2회를 추가로 지불하고, 경합 시 `tryLock` 대기(최대 5초)가 Tomcat 스레드를 붙잡습니다. 재고가 다 떨어진 뒤에도 모든 요청이 락 큐를 통과해야 409를 받을 수 있었습니다. 병목이 이동했을 뿐 사라지지 않은 것입니다.

### 2.2 Redisson 빈 이름 충돌

전략 빈을 `@Component("redisson")`으로 등록하자 컨텍스트가 죽었습니다:

```
BeanDefinitionOverrideException: Invalid bean definition with name 'redisson'
  defined in [RedissonAutoConfigurationV2.class]
```

redisson-spring-boot-starter가 자기 클라이언트 빈을 같은 이름으로 등록하고 있었습니다. 전략 빈에 `Strategy` 접미사를 붙여 해결했지만, "라이브러리 스타터의 빈 네임스페이스를 침범하지 말 것"이라는 교훈을 남겼습니다.

### 2.3 폭주 구간에서 모니터링이 눈을 감았다

①/② 런의 Prometheus 지표가 이상했습니다 — `tomcat_busy_max=2` (실제로는 200 포화였을 텐데). 원인: **스크레이프 요청도 다른 요청과 같은 TCP 백로그에 줄을 선다**. 폭주 구간에 스크레이프가 연결 거부를 당해, 남은 샘플은 평온한 구간뿐이었습니다. 서버가 살아있던 ③ 런만 서버측 지표가 유효했습니다. 이때부터 비교의 기준을 k6 클라이언트측 수치 + DB/Redis 카운트로 통일했습니다. "관측 시스템도 장애 도메인 안에 있다"는 것을 직접 겪은 셈입니다.

## 3. 해결 과정 — 3전략 비교표

| 선택지 | 장점 | 단점 | 채택 |
|---|---|---|---|
| ① MySQL 비관적 락 | 구현 단순, 정합성 확실 | 행 락 직렬화 → 스레드/풀/TCP 연쇄 붕괴 | ✗ (baseline) |
| ② Redisson 분산락 | 범용 패턴, 다중 인스턴스 대응 | 락 왕복+대기가 스레드 점유, **①보다 처리량 절반** | ✗ |
| **③ Redis Lua 원자 스크립트** | 락 없이 검사+차감+등록 원자 실행, 재고 소진 후엔 DB 무접촉 | Redis 장애 시나리오 설계 필요 (Phase 3+) | **✓** |

③의 핵심은 8줄짜리 Lua 스크립트입니다:

```lua
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then return 'DUPLICATE' end
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock <= 0 then return 'SOLD_OUT' end
redis.call('DECR', KEYS[1]); redis.call('SADD', KEYS[2], ARGV[1])
return 'OK'
```

여기에 Caffeine 메타 캐시(오픈시각/상태만, 잔여 수량은 절대 캐시하지 않음)를 앞단에 두어 판정 경로에서 DB를 완전히 제거했습니다. 중복 방어는 3중입니다: Lua의 SISMEMBER → DB `uk_event_user` 유니크 제약 → Testcontainers 동시성 테스트(300명→정확히 100장)가 CI에서 상시 검증.

## 4. 결과

동일 스파이크(100→5,000rps, 재고 10,000) 기준:

| 지표 | ① 비관적 락 | ② Redisson | ③ Lua |
|---|---|---|---|
| 정상 처리 (201/409) | 109,823 (14.4%) | 54,350 (7.1%) | **336,727 (44.3%)** |
| 발급(201) 응답 중앙값 | 12,042ms | 9,421ms | **34ms** (354×) |
| 발급(201) p95 | 44.4s | 34.2s | **1.42s** |
| TCP 연결 거부 | 652K | 707K | 423K |
| 정합성 (초과/중복/Redis-DB 대사) | 0/0/— | 0/0/일치 | 0/0/일치 |

③에서도 TCP 거부 42만 건이 남았습니다. 서버측 지표가 원인을 가리킵니다: `hikaricp.connections.pending` 최대 160 — **발급 이력 INSERT가 아직 동기**라서 발급 성공 구간에서는 여전히 스레드가 DB에 블록됩니다. 이것이 다음 Phase의 동기입니다.

## 5. 배운 점

- 분산락을 도입하면 빨라질 거라 기대했지만 실측하니 비관적 락보다 정상 처리가 절반이었고, 원인이 락 왕복+대기의 스레드 점유임을 확인해 "락을 옮기는 것"과 "락을 없애는 것"은 완전히 다른 개선임을 수치로 배웠습니다.
- Lua 원자 스크립트로 판정 경로에서 DB와 락을 모두 제거했더니 발급 응답 중앙값이 12초에서 34ms로 354배 개선되었고, 그 상태에서 초과·중복 발급 0건을 CI 동시성 테스트로 보장했습니다.
- 폭주 구간에서 Prometheus 스크레이프가 TCP 거부에 막혀 서버 지표가 거짓말을 하는 것을 발견했고, 부하 실험의 1차 근거는 클라이언트측(k6) 수치 + 저장소 카운트로 삼아야 함을 배웠습니다.

## 6. 다음 Phase 예고

Lua 전환 후 남은 병목은 명확합니다: **동기 DB INSERT (hikari pending 160)와 동기 알림 호출**. Phase 3에서는 먼저 mock-notify에 3초 지연을 주입해 동기 호출이 발급 API 전체를 무너뜨리는 장애를 재현하고, Redis Stream + 워커로 기록/알림을 임계 경로에서 밀어낸 뒤, 같은 스파이크에서 p99가 흔들리지 않음을 보입니다. 그다음이 Tomcat vs Netty입니다.
