# Phase 1 회고 — 정직한 MVP: 처참한 baseline을 수치로 소유하기

> 2026-07-21 · 커밋 범위: `9d16f71`~`c508174` · 근거 리포트: [baseline](../reports/phase1-baseline.md) · [HikariCP](../reports/phase1-hikari-experiment.md) · [explain](../reports/phase1-explain-tuning.md)

## 1. 배경과 목표

Phase 1의 목표는 역설적입니다 — **의도적으로 소박하게 만들어서, 잘 무너지는 모습을 정확히 측정하는 것**([로드맵 5장](../coupon-system-roadmap.md)). 발급은 `SELECT ... FOR UPDATE` 비관적 락, 이력 조회는 인덱스 없이 시작했습니다. 목표 수치(p99 200ms, 5,000rps)는 이 Phase에서 달성하는 것이 아니라, 얼마나 못 미치는지를 기록하는 기준선입니다.

## 2. 문제 상황

### 2.1 5,000rps 스파이크: 실패의 95%는 애플리케이션 밖에서 일어났다

스파이크 테스트(100→5,000rps)의 결과는 예상대로 처참했지만, **무너지는 위치**가 흥미로웠습니다. 585,177개 요청 중 99.15%가 실패했는데, 그중 552,791건이 HTTP 에러가 아니라 `connectex: No connection could be made` — **TCP 연결 거부**였습니다. 서버에 도달한 요청은 6.4%(37,157건)뿐.

Grafana가 사슬을 보여줬습니다: Tomcat 스레드 200/200 포화 → 그중 178개가 HikariCP 풀(20) 대기 → 풀을 잡은 20개는 전부 이벤트 행 하나의 락 대기. 스레드가 바닥나니 accept 백로그가 넘치고, OS가 연결을 거부하기 시작한 것입니다. "DB가 느리다"가 아니라 **락 직렬화가 스레드 고갈로, 스레드 고갈이 TCP 거부로 전파**되는 구조였습니다.

그 와중에 발급은 정확히 10,000건 — 초과 0, 중복 0. 비관적 락은 정확하지만, 정확함의 대가가 전부였습니다.

### 2.2 이력 조회: 단건 110ms짜리 쿼리의 대기열 붕괴

51만 행에서 인덱스 없는 이력 조회는 `type=ALL` 풀스캔 + filesort, 단건 110ms였습니다. "110ms면 쓸만한데?"가 함정입니다 — 300rps로 들어오는 순간 DB CPU가 포화되고, 달성 처리량 43rps, 응답 중앙값 **43.7초**로 붕괴했습니다. 단건 성능과 부하 성능은 완전히 다른 문제였습니다.

### 2.3 도구의 배신: 부하테스트를 죽인 것은 서버가 아니라 PowerShell이었다

측정 자동화 스크립트가 두 번 연속 부하 도중 죽었습니다. 원인은 서버가 아니라 PowerShell 5.1 — mysql 클라이언트의 비밀번호 경고, k6의 요청 실패 경고 같은 **stderr 한 줄을 NativeCommandError로 승격**시켜 `$ErrorActionPreference=Stop`과 만나 스크립트를 중단시켰습니다. `MYSQL_PWD` 환경변수로 경고 자체를 없애고, k6 구간만 에러 허용으로 완화해 해결했습니다.

## 3. 해결 과정

### 3.1 HikariCP 풀 크기 — 대기를 어디에 세울 것인가

동일 부하(1,000rps × 2분)로 풀 크기만 바꾼 실험:

| 선택지 | 장점 | 단점 | 채택 |
|---|---|---|---|
| pool=5 | DB 부담 최소 | 풀 대기 194 스레드, 중앙값 1,130ms (기아) | ✗ |
| **pool=20** | 중앙값 **91ms**, 서버 p99 0.23s — 5~12배 우위 | — | **✓** |
| pool=100 | 풀 대기는 감소(99) | 커넥션 100개가 같은 행 락을 경합, 중앙값 1,077ms로 재악화 | ✗ |

풀을 키우면 대기가 사라지는 게 아니라 **풀 앞에서 DB 안으로 이동**했습니다. 12코어 머신에서 최적점 20은 HikariCP 권고(2×코어)와 대략 일치합니다. 세 조건 모두 발급(201) 중앙값은 ~18초로 동일했는데, 이것이 핵심 증거입니다 — 근본 병목(락 직렬화)은 풀 튜닝으로 해결되지 않습니다.

### 3.2 인덱스 설계 — 커버링을 하지 않기로 한 이유

| 선택지 | 장점 | 단점 | 채택 |
|---|---|---|---|
| 인덱스 없음 | 쓰기 최속 | 풀스캔 51만 행, 43rps 붕괴 | ✗ (의도된 시작점) |
| **(user_id, issued_at)** | range 접근 + filesort 제거, Backward index scan | 행당 PK 룩업 잔존 | **✓** |
| 전 컬럼 커버링 | PK 룩업도 제거 | JPA가 전 컬럼 SELECT라 인덱스에 전 컬럼 필요 — 비대화 | ✗ (사용자당 ≤10행이라 이득 미미) |

"커버링 인덱스가 항상 답"이 아니라, **SELECT 컬럼(JPA 엔티티 조회)과 행 분포(사용자당 행 수)를 보고 멈출 지점을 정하는 것**이 실제 판단이었습니다.

## 5. 배운 점

- 5,000rps 스파이크에서 실패율 99.15%를 확인했고, 원인을 추적하니 Tomcat 스레드 200개 전부가 단일 행 락 → 커넥션 풀 순으로 블록되어 TCP 백로그까지 넘친 것이었으며, 실패의 95%가 애플리케이션 코드 밖(TCP 거부)에서 발생함을 확인했습니다.
- 커넥션 풀을 5→20→100으로 바꿔 측정했더니 20에서 중앙값 91ms로 최적이었고, 100에서는 오히려 1초대로 악화되어 "풀 확대는 대기 위치를 옮길 뿐"임을 수치로 증명했습니다.
- 단건 110ms 풀스캔 쿼리가 300rps 부하에서는 중앙값 43초로 붕괴하는 것을 보고, `(user_id, issued_at)` 인덱스로 중앙값 5.73ms·처리량 300rps를 달성했습니다 — 단건 성능이 아니라 부하 성능으로 판단해야 함을 배웠습니다.
