## 근본 원인

증상(전면 지연)만 보면 "DB가 느리다"로 오판하기 쉽다. 가설 검증:

1. **가설 A: DB 과부하** → 반증: tcpdump에서 신규 연결 시도가 없고(SYN 1건), 기존 커넥션 12개의 트래픽뿐. DB에 도달하는 동시 쿼리는 최대 2개(풀 크기)로 오히려 한가하다.
2. **가설 B: 풀 대기** → 입증: `hikaricp.connections.pending=197` — Tomcat 스레드 200개 중 197개가 커넥션 반납을 기다리는 큐에 있었다. 애플리케이션 로그에는 아무 에러도 없다(타임아웃 30s에 도달하기 전이므로) — **로그 없는 장애**.

근본 원인: 풀 크기(2)가 동시 수요(행 락 직렬화로 커넥션 점유 시간이 길어진 200 동시 요청)에 비해 극단적으로 작음. 증거 파일: `k6-results/drill1-evidence.txt`, `drill1-traffic-sample.txt`.

## 배운 점

- 커넥션 풀 고갈은 DB 지표가 아니라 **애플리케이션 쪽 대기 지표 + 패킷의 부재(신규 연결 없음)**로 증명된다.
- 풀 대기는 타임아웃 전까지 로그가 없다 — 지연 + 무로그 조합의 1순위 용의자.
