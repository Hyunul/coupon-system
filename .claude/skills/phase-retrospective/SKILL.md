# Phase 회고 작성 (블로그 포스팅용)

Phase 종료 시 `docs/retrospectives/phase-<N>-<슬러그>.md` 로 회고 글을 작성한다.
파일명 예: `phase-0-기반공사.md`, `phase-1-정직한-mvp.md`, `phase-2-redis-3전략.md`

## 1단계 — 증거 수집 (글쓰기 전 필수)

1. `git log --oneline`으로 해당 Phase의 커밋 범위를 확인한다.
2. `docs/reports/`에서 해당 Phase의 실험 리포트를 모두 읽는다. **글에 들어가는 모든 수치는 여기(또는 k6-results/의 summary JSON)에서 인용한다.**
3. 해당 Phase에서 실제로 겪은 문제(트러블슈팅)를 커밋 메시지·리포트·대화 기록에서 수집한다.
4. Grafana 스크린샷이 리포트에 있으면 경로를 확인해 글에서 참조한다.

## 2단계 — 글 구조 (이 순서 고정)

1. **배경과 목표** — 이 Phase에서 무엇을 왜 했는가. 로드맵([docs/coupon-system-roadmap.md](../../docs/coupon-system-roadmap.md))의 해당 절을 인용하고 목표 수치를 명시.
2. **문제 상황** — 실제로 부딪힌 문제를 증상 → 원인 규명 과정 순서로. 측정값·로그·에러 메시지를 그대로 인용한다. 문제가 여러 개면 소제목으로 나눈다.
3. **해결 과정** — 시도한 것들과 각 판단의 근거. **대체 가능한 기술과의 비교표 필수**:

   | 선택지 | 장점 | 단점 | 채택 여부와 이유 |
   |---|---|---|---|

   실제로 검토하거나 구현한 대안만 넣는다 (예: 재고 차감 3전략, WireMock vs Node mock 서버).
4. **결과** — before/after 수치 표. 측정값 없는 주장은 쓰지 않는다.
5. **배운 점** — 면접에서 그대로 말할 수 있는 형태의 문장 2~3개 ("~를 확인했고, ~를 했더니, ~가 되었다" 구조).
6. **다음 Phase 예고** — 이번 결과가 만들어낸 다음 질문.

## 3단계 — 규칙

- **모든 수치는 실측값만.** 측정하지 않은 항목은 지어내지 말고 "미측정"으로 남긴다.
- 코드 스니펫은 핵심만 10줄 이내로 인용한다.
- 톤: 기술 블로그. "~했습니다" 체, 과장 없이, 실패·한계도 그대로 기록 (균형 잡힌 스토리가 더 강하다).
- 글 길이: 1,500~3,000자 내외 (너무 길면 분할 제안).

## 4단계 — 마무리 갱신

회고 작성 후 반드시:
1. [README.md](../../README.md) 진행 상태 표에서 해당 Phase를 ✅로 갱신하고 회고 링크를 건다.
2. [CLAUDE.md](../../CLAUDE.md)의 "현재 상태" 절을 다음 Phase 기준으로 갱신한다.
3. 회고 파일과 갱신 파일을 함께 커밋한다 (예: `[P1] Phase 1 회고 작성`).
